# Тестовое задание: Мини-приложение "PromoCode Manager"

**Длительность:** 8-10 часов
**Результат:** Публичный Git-репозиторий с готовым проектом

---

## Цель

Создать fullstack приложение для управления промокодами с аналитикой.

Задание проверяет понимание **CQRS-подхода (Command Query Responsibility Segregation)** — архитектуры, в которой пути записи и чтения данных разделены между разными хранилищами:

- **Command (запись):** мутации данных идут в основную БД (MongoDB)
- **Query (чтение для таблиц):** все таблицы на фронтенде читают данные из аналитической БД (ClickHouse)
- **Синхронизация:** при каждой мутации в MongoDB данные реплицируются в ClickHouse

**Пользователи:**
- Базовая информация (email, имя, телефон)
- Пароль (хранится в хэшированном виде)
- Статус активности

**Промокоды:**
- Уникальный код (например, "SUMMER2024")
- Процент скидки
- Лимиты использования (общий и на пользователя)
- Срок действия (опционально: дата начала и окончания)
- Статус активности

**Заказы (упрощённые):**
- Принадлежность пользователю (для агрегации статистики)
- Сумма заказа
- Применённый промокод (опционально)
- Дата создания
- Реализация может быть моковой — главное, чтобы были данные для аналитики

**Использования промокодов:**
- Какой промокод, кем и к какому заказу применён
- Рассчитанная сумма скидки
- Дата использования

### Функциональность

**Авторизация и регистрация:**
- Регистрация нового пользователя (email, пароль, имя, телефон)
- Вход по email и паролю
- JWT токены для аутентификации запросов
- Защита API endpoints (только авторизованные пользователи)

**CRUD операции:**
- Создание, чтение, обновление, деактивация пользователей и промокодов

**Заказы:**
- Создание заказа (сумма, принадлежность пользователю). Заказ существует **независимо** от промокода
- Просмотр своих заказов

**Применение промокода:**
- Отдельный endpoint для применения промокода к **уже существующему** заказу
- Это два разных действия: сначала пользователь создаёт заказ (`POST /orders`), затем применяет промокод к этому заказу (`POST /orders/:id/apply-promocode`)
- Валидация: существование заказа, существование и активность промокода, лимиты, срок действия, промокод ещё не применён к этому заказу
- Расчёт и сохранение скидки

**Таблицы на фронтенде (данные из ClickHouse):**
- Таблица пользователей с агрегированной статистикой (заказы, суммы, использованные промокоды)
- Таблица промокодов с метриками эффективности (использования, выручка, уникальные пользователи)
- История использований промокодов

**Требования к таблицам:**
- Server-side пагинация, сортировка, фильтрация по колонкам
- Глобальный фильтр по диапазону дат (важно для производительности ClickHouse)
- Пресеты дат: "Сегодня", "Последние 7 дней", "Последние 30 дней", "Произвольный"
- Загрузка всех записей в память на клиенте (`findAll()` без лимита) — **недопустима**

## Архитектура данных (CQRS)

Это **ключевой аспект задания**. Приложение должно реализовать разделение ответственности между базами данных:

- **MongoDB** — источник истины, CRUD операции, валидация бизнес-правил
- **ClickHouse** — аналитика, агрегаты, **все таблицы на фронтенде читают данные отсюда**
- **Redis** — кэширование, сессии, блокировки

### Роль Redis

Redis — **обязательный** компонент архитектуры. Примеры использования (реализуйте хотя бы два):

- **Кэширование** аналитических запросов к ClickHouse (результаты таблиц с коротким TTL)
- **Distributed lock** при применении промокода — предотвращение race condition, когда два запроса одновременно применяют один промокод и превышают лимит
- **Хранение сессий** или refresh-токенов
- **Rate limiting** на критичных endpoint'ах (login, apply-promocode)

Redis должен присутствовать в Docker Compose и использоваться в коде бэкенда. Просто добавить Redis в docker-compose без реального использования — недостаточно.

### Таблицы в ClickHouse

Все сущности из MongoDB должны быть **продублированы** в ClickHouse в денормализованном виде. Минимальный набор таблиц:

| Таблица ClickHouse | Содержимое                                                              |
| ------------------- | ----------------------------------------------------------------------- |
| `users`             | Данные пользователей (имя, email, телефон, дата создания, статус и пр.) |
| `promocodes`        | Данные промокодов (код, скидка, лимиты, даты, статус и пр.)             |
| `orders`            | Заказы (сумма, промокод, пользователь, дата и пр.)                      |
| `promo_usages`      | История использований промокодов                                        |

Данные в ClickHouse должны быть **самодостаточными** — аналитические запросы не должны обращаться к MongoDB. Например, таблица `promo_usages` должна содержать не только `userId`, но и имя/email пользователя; не только `promocodeId`, но и код промокода.

### Синхронизация MongoDB → ClickHouse

При каждой мутации в MongoDB (создание, обновление, деактивация) данные должны реплицироваться в ClickHouse. Конкретный механизм синхронизации — решение за вами (sync-on-write, event-driven, periodic batch и т.д.).

### Примеры data flow

**Пример 1: Создание промокода**
1. `POST /promocodes` → валидация DTO → запись в MongoDB
2. После записи → синхронизация в таблицу `promocodes` ClickHouse
3. Таблица промокодов на фронте → `GET /promocodes?page=1&pageSize=10&sortBy=createdAt` → запрос к ClickHouse

**Пример 2: Создание заказа и применение промокода (два отдельных действия)**
1. `POST /orders` → создание заказа (сумма, пользователь) → запись в MongoDB → синхронизация в ClickHouse
2. `POST /orders/:id/apply-promocode` → валидация промокода → обновление заказа (скидка) + запись использования в MongoDB → синхронизация `orders` и `promo_usages` в ClickHouse
3. Аналитическая таблица → агрегирующий запрос к ClickHouse (GROUP BY, SUM и пр.)

**Пример 3: Таблица пользователей с аналитикой**
1. Фронтенд запрашивает `GET /analytics/users?page=1&pageSize=10&sortBy=totalSpent&dateFrom=...`
2. Бэкенд выполняет запрос к ClickHouse с JOIN/GROUP BY
3. Ответ содержит готовые данные с именами, email, метриками — **без дополнительных запросов к MongoDB**

### Типичные ошибки (антипаттерны)

- Хранить в ClickHouse только ID без человекочитаемых данных (имя, email, код промокода) — аналитика будет показывать ObjectId вместо имён
- Иметь единственную таблицу в ClickHouse (например, только `promo_usages`) — недостаточно для полноценной аналитики
- Строить аналитику через запросы к MongoDB, а ClickHouse использовать "для галочки"
- Загружать все записи через `findAll()` и делать пагинацию/сортировку на клиенте
- Совмещать создание заказа и применение промокода в одном endpoint — это разные бизнес-операции

## Критерии оценки

Оценка производится по чеклисту. Каждый пункт оценивается по трёхуровневой шкале:

- **0** — не реализовано или не работает
- **1** — удовлетворительно (базовая реализация, работает)
- **2** — отлично (продуманная реализация, edge-cases учтены)

Это позволяет двум разным проверяющим прийти к одинаковому результату, при этом различая "сделано кое-как" и "сделано хорошо".

> **Примечание:** Модули Users и Orders оцениваются только на базовую работоспособность. Основной фокус — промокоды и аналитика.
